## Context

基于 gemini.md 文档描述的 AI Agent 协作平台，需要创建一个纯前端演示 demo 来展示核心交互流程。该 demo 不涉及后端逻辑，仅通过 Mock 数据展示平台的核心概念和交互设计。

**技术栈选择：**
- React 18 + TypeScript - 主流前端框架，类型安全
- Vite - 快速构建工具
- TailwindCSS - 原子化 CSS，快速构建 UI
- Recharts - 数据可视化（成就值雷达图、权重饼图）
- Lucide React - 图标库

**项目定位：**
- 演示性质，验证概念可行性
- 专注于交互体验展示
- 可独立运行，无需后端

## Goals / Non-Goals

**Goals:**
- 创建可独立运行的 React 前端演示项目
- 实现任务发布界面，展示自然语言输入和技能权重自动分配
- 实现任务匹配界面，展示多维度成就值匹配算法
- 实现 Skill 商城界面，展示共创持股和收益分配
- 实现成就值可视化，展示技能图谱雷达图

**Non-Goals:**
- 不实现真实后端 API 对接
- 不实现用户认证和授权
- 不实现真实的 OpenClaw 接入
- 不实现数据库存储
- 不实现智能合约或区块链功能

## Decisions

### 1. 单页面应用 vs 多页面应用

**决策：** 采用单页面应用 (SPA) 架构，使用 React Router 进行路由管理。

**理由：**
- Demo 规模适中，SPA 更简洁
- 页面间状态共享更方便（如用户成就值、任务数据）
- 用户体验更流畅

**替代方案：** 多页面应用 - 需要额外的状态管理，增加复杂度。

### 2. 状态管理方案

**决策：** 使用 React Context + useReducer，不引入 Redux/Zustand。

**理由：**
- Demo 规模不需要复杂的 global state
- 减少依赖，保持项目轻量
- Context 足够处理演示数据流

### 3. 数据存储方案

**决策：** 使用浏览器 localStorage 存储所有应用数据，首次加载时初始化样例数据。

**理由：**
- 无需后端即可实现数据持久化
- 用户刷新页面后数据不丢失
- 可以演示数据的增删改查完整流程
- 支持用户自定义数据（如发布新任务）

**实现方式：**
- 创建 `storage/` 目录封装 localStorage 操作
- 应用首次启动时检测 localStorage 是否为空
- 如果为空，写入预定义的样例数据（任务、用户、Skill 等）
- 提供"重置数据"功能，方便恢复初始状态

### 4. Mock 数据策略

**决策：** 使用独立的 `mock/` 目录存放初始化样例数据模板。

**理由：**
- 数据结构清晰，易于修改和扩展
- 样例数据与存储逻辑分离
- 可以展示不同场景的数据（如不同类型的任务）

### 5. 组件组织方式

**决策：** 按功能模块组织组件：`components/task/`, `components/matching/`, `components/marketplace/`, `components/achievement/`

**理由：**
- 与 specs 中的 capabilities 对应
- 便于定位和维护
- 符合功能边界的自然划分

## Risks / Trade-offs

**[Demo 与生产环境的差距]** → **缓解：** 代码结构保持清晰，为后期接入真实 API 预留接口

**[Mock 数据可能误导]** → **缓解：** 在 UI 中明确标注"演示模式"，关键逻辑添加注释说明

**[前端技术选型可能不是最优]** → **缓解：** Demo 目的是验证概念，技术栈可调整

## Migration Plan

3. 独立运行验证
